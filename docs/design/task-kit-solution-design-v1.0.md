## 2. 目的と設計対象
- 解決したい課題:
  - CLI によって、エージェント/プロンプト/スキル/テンプレートを対象ディレクトリへ安全に展開できる。
  - 展開済みキットを使い、チャットコマンド起点でタスク遂行の主要業務フローを再現可能にする。
- 設計対象(含む):
  - CLI による配布物展開
  - タスク作成/修正/アーカイブ
  - プラン作成/修正/実行
  - レビュープロンプト生成/レビュー実行支援
  - 報告書作成/修正
  - 課題管理、気づき/ログ取得、リスク検出
  - 引継ぎ資料作成
  - Slack/Teams 向けリード文作成(投稿文生成まで)
  - 上記を支えるエージェント、プロンプト、スキル、テンプレートの提供
- 設計対象外(含まない):
  - Slack/Teams への自動投稿
  - 外部SaaS との双方向同期
  - モデル学習/ファインチューニング/推論基盤の新規開発
  - 成果物最終承認の完全自動化
  - ワークスペース外の運用手順策定
- 成功条件:
  - `task-kit init` 実行で、現在のワークスペースに `.github` と `.task-kit` へ必要資産を展開できる。
  - `task-kit init --copilot` は `task-kit init` と同一動作の別表記として扱う。
  - `/task-kit.new-task <task title>` 実行で、日付階層を持つタスクディレクトリと初期ファイル群を生成できる。
  - 主要ユースケースが手順化され、再現実行できる。
  - 禁止事項(タスクキット利用事実の混入禁止、機密情報保護)を出力設計に組み込む。
  - CLI エラー受け入れ基準と終了コード(0-7)を操作導線に対応付ける。

## 4. 設計方針
- 方針1: 既存構成優先・最小変更
  - このリポジトリではテンプレート原本のみを管理し、CLI は配布専用に分離する。
- 方針2: 禁止事項を機能横断の制約として実装
  - すべての成果物生成フローに、出力前チェック(禁止語/機密情報)を組み込む。
- 方針3: CLI は失敗時に利用者の次アクションが即判断できる設計
  - エラー分類、終了コード、修正ガイドを一体化して提示する。
- 方針4: `.github` は最小構成、運用資産は `.task-kit` に集約
  - Copilot 実行に必要な入口資産のみ `.github` 配下へ置き、タスク運用の実体は `.task-kit` と `tasks` に置く。
- 既存構成との整合方針:
  - docs/design は設計一次情報の置き場として維持。
  - docs/templates は入力テンプレートの一次情報として維持。
  - collaboration 配下は索引専用とし、運用本文は重複させない。
- 互換性上の注意点:
  - 既存テンプレートの項目名を変更しない。
  - CLI の終了コード体系(0-7)を固定し、追加時は後方互換ルールを別途定義する。

## 5. 機能分解
| 機能 | 責務 | 入力 | 出力 | 備考 |
|---|---|---|---|---|
| F1. 配布物展開(init) | `.github` と `.task-kit` へキット一式を展開 | 展開先パス、上書き方針、`--copilot` | 配置結果、終了コード | 競合時は確認または失敗 |
| F2. タスク起票(new-task) | タスク作業領域の新規作成 | `/task-kit.new-task <title>` | `tasks/YYYY/MM/DD/NNN-slug` 一式 | 日次連番とslugを自動採番 |
| F3. タスク修正 | タスク定義の更新 | task.md、追加要件 | 更新済み task.md | 変更履歴を残す |
| F4. プラン作成/修正/実行 | 実行計画の構築・更新と実行記録 | plan.md、依存、リスク前提 | plan.md、records | 実行前後で差分記録 |
| F5. レビュープロンプト生成 | レビュー観点を定義した入力の生成 | 対象成果物、観点、制約 | レビュープロンプト | 禁止事項チェック対象 |
| F6. 報告書作成/修正 | 実行結果の報告の標準化 | 実行ログ、課題、判断 | outputs 配下成果物 | 出典と判断根拠を明示 |
| F7. 課題管理/助言 | 課題登録、状態更新、解決助言 | issue.md、records/findings.md | 課題更新、助言メモ | 追加と更新を分離 |
| F8. リスク検出 | 想定リスクとブロック課題の提示 | タスク/プラン内容、実行状況 | リスク一覧、ブロック一覧 | 実行前/実行中で再評価 |
| F9. 引継ぎ資料作成 | 次担当者向け状態の整理 | handoff.md、未完了事項 | handoff.md 更新 | 再開手順を必須項目化 |
| F10. Slack/Teams リード文作成 | 投稿用先頭文のテンプレート生成 | 報告要点、宛先、トーン | リード文テンプレート | 自動投稿は対象外 |
| F11. 気づき/ログ取得 | 判断と学習事項の収集 | records/scratchpad.md、レビュー結果 | findings 更新、ログ | 機密情報を除外 |

### 5.1 CLI コマンドカタログ(現時点)
CLI から実行可能とするコマンドを以下に固定する。`task-kit init` と、`task-kit init --copilot` は同一動作の別表記として扱う。

| コマンド | 目的 | 主入力 | 主出力 | 主更新対象 |
|---|---|---|---|---|
| `task-kit init` | キット配布物を現在のワークスペースへ展開 | 上書き方針、`--copilot`(任意・指定時も同一動作) | 展開結果、終了コード | `.github/`, `.task-kit/` |

補足: `--copilot` は省略可能で、指定しても指定しなくても同一の既定動作とする。 他のオプション（例えば `--claude` のような）を作成することを想定した実装とする。

### 5.2 チャットコマンドカタログ(現時点)
VS Code チャット入力から実行可能とするコマンドを以下に固定する。

| コマンド | 目的 | 主入力 | 主出力 | 主更新対象 |
|---|---|---|---|---|
| `/task-kit.new-task` | 新規タスク作業領域を作成 | タスクタイトル | `tasks/YYYY/MM/DD/NNN-slug` パス | `task.md`, `plan.md`, `issue.md`, `handoff.md` |
| `/task-kit.task-update` | タスク定義を更新 | タスクパス、変更要求 | 更新済み task.md、差分要約 | `task.md`, `records/findings.md` |
| `/task-kit.plan-update` | 実行計画を更新 | タスクパス、依存、制約 | 更新済み plan.md | `plan.md`, `records/scratchpad.md` |
| `/task-kit.task-execute` | プラン実行支援と記録更新 | タスクパス、対象ステップ | 実行ログ、進捗更新 | `plan.md`, `records/*`, `outputs/` |
| `/task-kit.review` | レビュー実行支援 | タスクパス、レビュー対象、観点 | 指摘一覧、修正提案 | `records/findings.md`, `outputs/` |
| `/task-kit.issue-consult` | 課題の解決助言 | タスクパス、課題IDまたは課題記述 | 助言、対応案、優先度提案 | `issue.md`, `records/findings.md` |

### 5.3 チャットコマンド実行エージェント方針(確定)
チャットコマンドの実行責務は次の2エージェントへ分離する。

- `task-kit.task`:
  - 対象コマンド: `/task-kit.new-task`, `/task-kit.task-update`, `/task-kit.plan-update`, `/task-kit.task-execute`, `/task-kit.issue-consult`
  - 目的: コマンド間整合、タスク内整合、状態遷移整合を一貫して管理する。
- `task-kit.review`:
  - 対象コマンド: `/task-kit.review`
  - 目的: 実行系エージェントから独立したレビュー観点を維持する。


## 6. 境界と責務
- 画面/CLI:
  - CLI は入力検証、処理起動、終了コード返却を担当。
  - UI は持たず、VS Code チャットコマンド(`/task-kit.new-task`, `/task-kit.task-update`, `/task-kit.plan-update`, `/task-kit.task-execute`, `/task-kit.review`, `/task-kit.issue-consult`)とファイル操作を前提とする。
- API/外部接続:
  - 外部接続は AI 実行基盤利用のための通信に限定。
  - 外部SaaS との双方向同期は行わない。
- データ/状態管理:
  - 文書ファイルを正本とし、`tasks/YYYY/MM/DD/NNN-slug` 単位で状態を管理。
  - 状態遷移は「未着手/計画済み/進行中/完了」を最小セットとする。
- 権限/認可:
  - ローカルファイル書き込み権限とネットワーク利用権限を前提。
  - プラン制約(有料プラン必須)を起動時チェックで判定。
- 例外系/失敗時の扱い:
  - 終了コード0-7を固定し、各エラーに修正例を付与。
  - 予期しない例外は要約メッセージと最小ログ参照先のみ表示。

## 7. データとインターフェース方針
- 主なデータ構造:
  - タスクディレクトリ: `tasks/YYYY/MM/DD/NNN-slug`
  - 初期ファイル: `task.md`, `plan.md`, `issue.md`, `handoff.md`
  - 記録ファイル: `records/findings.md`, `records/scratchpad.md`
  - 初期ディレクトリ: `references/`, `outputs/` (`.gitkeep` を配置)
- 保存/更新の単位:
  - 1タスク1ディレクトリを基本単位とし、配下文書を更新する。
  - 日次フォルダ内で3桁連番(`001-...`)を採番する。
- インターフェース契約:
  - new-task 入力: タイトル必須、空文字は終了コード1。
  - new-task 出力: 新規タスクパスを標準出力で返却。
  - task-update 入力: タスクパスと変更要求必須。未指定は終了コード1。
  - plan-update 入力: タスクパスと計画更新内容必須。未指定は終了コード1。
  - task-execute 入力: タスクパス必須。対象ステップ未指定時は未完了ステップを既定対象とする。
  - review 入力: タスクパスとレビュー対象必須。レビュー観点省略時は標準観点を適用。
  - issue-consult 入力: タスクパスと課題記述必須。課題IDがあれば優先参照する。
  - CLI 入力: 必須引数不足は終了コード1。
  - パス/権限異常: 終了コード2。
  - ネットワーク異常: 終了コード3。
  - プラン制約違反: 終了コード4。
  - 競合: 終了コード5。
  - 内部例外: 終了コード6。
  - キャンセル: 終了コード7。
- バリデーション方針:
  - 必須項目、禁止事項、機密情報混入、状態遷移不整合をチェック対象にする。
  - リスク検出は実行前(静的)と実行中(動的)の2段階で実施する。

## 8. 非機能前提
- 性能:
  - 定量SLOを置かず、主要ユースケースの完走性を優先。
- 可用性/運用:
  - Windows/macOS + VS Code 1.102以上を前提。
  - オフライン運用は対象外。初期導入と運用時にネットワーク前提。
- 監査/ログ:
  - 実行ログは最小限(時刻、機能、結果、参照先)を記録。
  - 調査可能性を確保しつつ、機密情報は記録しない。
- セキュリティ/権限制約:
  - 機密情報、資格情報、個人情報を成果物・ログに出力しない。
  - タスクキット利用事実を成果物に混入しない。

## 11. 実装引き継ぎ事項
- 実装開始条件:
  - 本設計書と要件書の対応表を実装計画に転記済みであること。
  - CLI 終了コード0-7の取り扱いに実装者間の解釈差がないこと。
- 先に確認すべき事項:
  - 既存テンプレートの項目互換性(項目削除なし)を確認。
  - 禁止事項チェックを共通処理化する配置を確認。
  - `new-task` 採番規約(日次3桁連番)と slug 変換規則を確定。
- 想定タスク分割:
  1. 配布元テンプレート整備(`templates/github`, `templates/.task-kit`)
  2. ビルド時コピー処理(`scripts/build-templates`)の実装
  3. CLI 基盤(入力検証、終了コード、共通エラー整形)
  4. `init` と `new-task` の実装
  5. 機能コマンド(F3-F11)の追加
  6. 禁止事項/機密情報チェック導入
  7. 運用ドキュメント整備(手順、受け入れ条件、例示)
- テスト観点:
  - 正常系: `init` 後に `new-task` で初期ファイル群が生成されること
  - 正常系: 主要ユースケース完走(作成→実行→レビュー→報告→引継ぎ)
  - 異常系: 終了コード1-7の再現、次アクション表示確認
  - 異常系: 同日連番の競合時に次番号で作成できること
  - 品質系: 禁止事項違反の未混入確認、機密情報非出力確認
  - 互換系: 既存テンプレートで破壊的変更がないこと
